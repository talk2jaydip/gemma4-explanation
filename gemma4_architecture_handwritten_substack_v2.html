Handwritten architecture guide

Gemma 4, drawn out

Most model announcements hand you a table and a handful of benchmark numbers. That is useful, but it does not explain the machine. This guide treats Gemma 4 like a system on a whiteboard: where the tokens enter, how image patches become soft tokens, why the context window does not explode immediately, and what "26B A4B" actually means when a single token is being processed.

The short version: Gemma 4 is a family, not one model. The small dense models are tuned for devices and include audio; the larger models push long-context text-plus-vision reasoning; the 26B A4B model is the sparse MoE member, where only about 3.8B parameters are active per token.

The Gemma 4 family, without the fog

Gemma 4 ships as four public sizes: E2B, E4B, 26B A4B, and 31B. Google describes the family as multimodal, with text and image input for all models, audio support on the smaller E2B and E4B models, and text output. The official card also names dense and MoE variants, context windows up to 256K tokens, a 262K-token vocabulary, and native system-role plus function-calling support.1

Memory note: Google's overview lists approximate inference weight memory at BF16/SFP8/Q4. For example, Q4 estimates are 3.2 GB for E2B, 5 GB for E4B, 15.6 GB for 26B A4B, and 17.4 GB for 31B. Those are weight-memory estimates, not the whole runtime footprint with KV cache, batching, or framework overhead.2

A token enters the machine

Gemma 4 is easiest to understand as a decoder-first multimodal transformer. Text tokens are looked up in an embedding table. Images are converted into soft visual tokens by a vision encoder. Audio, only on E2B and E4B, is converted into audio token embeddings before it joins the sequence. Then the combined sequence goes through decoder layers that repeatedly answer one question: "Given everything so far, how should this position be rewritten?"

Key mental picture: the model is not reading pixels and words in separate universes. After the encoders, it sees a single sequence of vectors with special image/audio/text positions. The decoder is where the modalities actually meet.

Step 3 - Tokenizer and vocabulary: the 262K-door

Before any attention happens, text has to become integers. Gemma 4 uses a 262,144-token SentencePiece vocabulary.1 That is huge by older standards: GPT-2 was 50,257; Llama 2 was 32,000. Bigger vocab means fewer tokens per string, which means longer effective context for the same 256K budget and less fragmentation on non-Latin scripts and code.

Heuristic: avg bytes-per-token tends to scale with log2(vocab). vocab=32K -> tokens per 1 KB English text ~ 260 vocab=128K -> tokens per 1 KB English text ~ 210 vocab=262K -> tokens per 1 KB English text ~ 185 So a 256K-context window holds roughly 1.4 MB of English text (vs ~1.0 MB on a 32K-vocab model at the same context).
Cost note: the embedding table is 262,144 * hidden_dim weights. That lives in the weight file, not the KV cache. It does NOT grow with context length. The trade is: bigger file, cheaper inference per token on average.

The decoder block: the part that does most of the thinking

The original Transformer gave us the broad recipe: self-attention mixes information across positions, then a position-wise feed-forward network edits each position independently.8 Gemma 4 keeps that lineage but makes modern choices around normalization, local/global attention, RoPE, and parameter efficiency.

1. Normalize before reading

RMSNorm keeps the residual stream numerically calm. Keras' Gemma 4 backbone notes four norms per decoder block: pre and post around attention, and pre and post around the feed-forward path.6

2. Build Q, K, V

The hidden vector is projected into queries, keys, and values. Query asks: "What am I looking for?" Key says: "What do I contain?" Value says: "What should I pass along?"

3. Read locally or globally

Most layers can be sliding-window layers; some are full/global. This gives a cheap short-range sketch plus occasional long-range consolidation.

4. MLP or MoE edits

Dense models use dense feed-forward blocks. The 26B A4B model uses a sparse expert path, so a token visits a small set of experts instead of all expert parameters.

Attention sketch: Q = X Wq, K = X Wk, V = X Wv scores = Q K^T weights = softmax(scores) output = weights V Classic Transformers scale by 1/sqrt(head_dim). Gemma 4 implementation docs note Q/K/V normalization and attention scaling = 1.0 in the Keras backbone, so the stability job is handled differently in that implementation.

Hybrid attention: the 256K-context escape hatch

A full attention layer over 256K tokens is expensive because every position can compare itself with every other position. Gemma 4's model card says the models interleave local sliding-window attention with full global attention, and that the final layer is always global. Public implementation docs describe a repeating local/global pattern; the 31B public config shows five sliding layers followed by one full layer across 60 layers.1 5 6

For a 256K context, N = 262,144 tokens. Full attention scores per head: N^2 = 262,144^2 = 68,719,476,736 scores Sliding-window scores with W = 1,024: N * W = 262,144 * 1,024 = 268,435,456 scores Reduction for that layer: 68,719,476,736 / 268,435,456 = 256x fewer score entries

That does not mean the model "ignores" long-range information. Think of local layers as scribbling in the margins near each paragraph. Global layers are the periodic table-of-contents pass, when distant sections can directly meet.

Step 6 - GQA: many askers, fewer answerers

Grouped-Query Attention (GQA) splits the trade between Multi-Head Attention (one KV per Q head, expensive) and Multi-Query Attention (one KV for all, cheap but lossy). Each KV head serves a group of Q heads.10 The 31B public config uses 32 query heads with 16 KV heads in sliding layers and 4 KV heads in global layers.5

KV cache bytes per token per layer, at head_dim = 256, BF16 (2 bytes): Full MHA (hypothetical 32 KV): 2 * 32 * 256 * 2 = 32,768 bytes Local GQA (16 KV): 2 * 16 * 256 * 2 = 16,384 bytes (50%) Global GQA (4 KV, dim 512): 2 * 4 * 512 * 2 = 8,192 bytes (25%) Savings compound across 60 layers and 262K tokens.
Why this is not just "a cache trick": with fewer KV heads, you also need less bandwidth to move them off HBM at each decode step. That is often the real bottleneck on GPUs, not arithmetic.

Step 7 - The 60-layer rhythm

Public 31B config uses a repeating pattern of 5 sliding local layers followed by 1 full global layer across 60 layers.5 The final layer is always global so the last prediction sees the full context.

60 layers / 6-long pattern = 10 full groups -> 50 sliding + 10 global. Share of KV memory at 256K context (per earlier table): Local layers: 1024 * 50 * 16384 bytes = 0.78 GiB ( 3.8% ) Global layers: 262144 * 10 * 8192 bytes = 20.0 GiB (96.2% ) -> global layers dominate KV memory even though they're only 1 in 6.

RoPE and p-RoPE: position as a rotation

RoPE, short for Rotary Position Embedding, rotates parts of the query and key vectors by an angle tied to token position. The useful trick is that the dot product between rotated vectors carries relative position information without adding a separate position vector to the token.9

Gemma 4 uses ordinary RoPE-style behavior for local/sliding attention and proportional RoPE for full/global layers in public configs. The intuition is simple: local layers only need precise nearby geometry; global layers need a position scheme that remains usable across very long context. For images, Gemma 4 also needs two-dimensional position information: left-right and up-down, not just "token number 42".3 5

KV cache math you can feel

When a decoder generates text, it stores keys and values from previous tokens so it does not recompute the whole prompt for every new word. That storage is the KV cache. Long context makes the cache large; sliding attention makes most layers only need a rolling window.

The following calculation uses public 31B config values: 60 layers, 32 query heads, 16 key-value heads in sliding layers, 4 global key-value heads with 512 global head dimension, BF16 at 2 bytes per number, and a public layer pattern with 10 global layers and 50 sliding layers.5

Local KV bytes per token per layer: 2 for K,V * 16 KV heads * 256 head_dim * 2 BF16 bytes = 16,384 bytes Naive full cache for all 60 layers: 262,144 tokens * 60 layers * 16,384 bytes = 240 GiB Hybrid-style estimate: Local layers: 1,024 window * 50 layers * 16,384 bytes = 0.78 GiB Global layers: 262,144 tokens * 10 layers * (2 * 4 * 512 * 2 bytes) = 20.0 GiB Total rough KV = 20.78 GiB

This is a teaching estimate. Real memory depends on runtime, quantized KV cache, padding, batching, paged attention, allocator overhead, and whether the framework keeps local-layer cache in a compact rolling form.

Vision: variable resolution instead of one forced square

Gemma 4's vision path is one of the cleaner design shifts: it can preserve image aspect ratio while controlling cost with a token budget. Hugging Face's Gemma 4 docs describe token budgets of 70, 140, 280, 560, and 1,120 soft tokens, with 280 as the default. They also state that the pre-pooling patch count is nine times the soft-token budget because 3x3 neighboring patches are averaged into one final visual token.3

Default image budget = 280 soft tokens Pre-pooling patches = 280 * 9 = 2,520 patches Each patch = 16 * 16 pixels One exact grid: 60 * 42 patches = 2,520 patches Image size: (60*16) by (42*16) = 960 by 672 pixels After 3x3 pooling: (60/3) * (42/3) = 20 * 14 = 280 visual tokens

Google's vision docs describe the tradeoff plainly: high budgets preserve more detail; low budgets reduce inference cost. That is why a document OCR prompt may deserve 560 or 1,120 tokens, while a quick scene caption can often live with 140 or 280.4

Practical rule: use the smallest image budget that still preserves the detail your question depends on. If the question is about a small label, a tiny chart axis, or a receipt line item, spend more visual tokens.

Step 11 - Audio, on E2B and E4B only

Only the smaller models take audio input. The design choice makes sense: E2B and E4B are the models built for device-local assistants, where you want voice transcription without a cloud round-trip. The 26B and 31B are rarely deployed in the same loop as a mic.1 7

Rough audio-token budget for a 30-second clip: 16 kHz waveform, 10 ms hop -> 100 mel frames per second 30 sec * 100 frames/sec = 3,000 mel frames After encoder downsampling (~4x): ~750 audio tokens So a 30-second voice note adds ~750 tokens to the sequence, leaving plenty of room in a 128K-context E4B for follow-up turns.

MoE: 26B A4B is not "a 4B model" and not "a 26B dense model"

The 26B A4B variant is a sparse Mixture-of-Experts model. The official model card lists 25.2B total parameters, 3.8B active parameters, 30 layers, 128 total experts, 8 active experts, and 1 shared expert.1 The ancestral idea goes back to sparse MoE layers: use conditional computation so capacity rises faster than per-token compute.11

Active-parameter ratio: 3.8B active / 25.2B total = 0.1508 = 15.1% Interpretation: The checkpoint can store a 25.2B-parameter expert library. A token only uses a routed subset plus the shared path. So per-token compute behaves much closer to the active size than to the total size.

The subtlety: routing is per token, not per whole prompt. In one sentence, the code-like token may route to different experts than a table header or a French phrase. That is why MoE can feel like many small specialists living inside one model.

Step 13 - PLE: per-layer notes for the small models

The "E" in E2B and E4B means effective parameters. These smaller models use Per-Layer Embeddings (PLE), where each decoder layer gets its own auxiliary embedding signal rather than relying only on the first token embedding.1

Hugging Face's implementation notes describe two PLE pieces: a token-identity lookup for the input IDs, and a context-aware projection from the input embeddings. The two are summed and scaled by 1/sqrt(2) before entering each decoder layer. For multimodal inputs where token IDs are unavailable, only the context-aware projection is used.3

PLE_layer_input = (token_identity + context_aware_projection) * (1/sqrt(2)) Variance check: if A and B are unit-variance and independent, Var(A + B) = 2, so multiplying by 1/sqrt(2) keeps unit variance. This is why the scaling is 1/sqrt(2), not 1/2.

That is a useful on-device move. Some parameters live in embedding lookups rather than heavier repeated compute. You can have a larger stored vocabulary-conditioned signal without making every forward pass behave like a much larger dense transformer.

Step 14 - Quantization: fitting Gemma 4 into real memory

Weight memory shrinks roughly in proportion to bit-width: BF16 is 2 bytes/param, SFP8 is 1 byte/param, and Q4 is 0.5 bytes/param with some overhead for scales and zero-points.2

Important: quantization shrinks WEIGHTS, not the KV cache. For 31B at 256K context (from earlier), KV cache alone ~ 20.8 GiB. If you quantize weights to Q4 (~17.4 GB) on a 24 GB card, you still need to leave room for KV + activations + framework. Rule of thumb for VRAM planning: weight_memory + kv_cache_at_max_context + 10-20% overhead

Step 15 - Prefill vs decode: why the first token feels slower

When you send a long prompt, the model runs two distinct phases. Prefill pushes every prompt token through every layer at once and fills the KV cache. Decode then appends one token per step, reading (not rebuilding) the cache.

Cost sketch (L = layers, d = hidden, N = prompt length): Prefill total ~ O(L * N * d^2) [all N tokens * linear proj] Prefill attn ~ O(L * N * W * d_h) [each token reads window] Decode per step ~ O(L * d^2 + L * N * d_h) [1 new Q, cache read] -> For a 10K-token prompt and a short answer, prefill can take longer than all decoding combined. -> Sliding windows make the N-dependent attention term grow linearly, not quadratically.
UX implication: Time-to-first-token is dominated by prefill. Tokens-per-second after that is dominated by decode + cache bandwidth. If you're seeing slow TTFT, shrink the prompt; if you're seeing slow streaming, profile bandwidth.

Step 16 - Function calling: a loop, not a secret "agent module"

Gemma 4 supports system-role prompting and function calling. Architecturally, the core model still predicts tokens. The difference is that instruction-tuned behavior and chat templates make the model produce structured text that a runtime can parse as a tool call, then append the tool result back into the conversation for the next decoding step.1 7

1. Schema
Give the model a tool name, fields, and constraints.
2. Decode
The model emits a structured call instead of normal prose.
3. Continue
The external result is inserted into context; the model reasons over it.
A single tool turn, on the wire: [system] You may call weather(city). [user] What's the weather in Paris? [model] <tool_call>weather(city="Paris")</tool_call> [runtime] parses, calls the function, gets "18C, clear" [tool] <tool_result>{"temp_c": 18, "sky": "clear"}</tool_result> [model] "It's 18 C and clear in Paris." -> same autoregressive decode; the runtime is what makes the loop.

The honest choice: start with the smallest model that preserves your needed modality, context length, and visual detail. Move up only when your eval set shows failures that are clearly capability failures, not prompting or retrieval failures.

Closing: what this means when you sit down to build

If you remember only three things:

  • Hybrid attention is the reason 256K context fits anywhere useful. Most layers are cheap local readers; a few are expensive global readers. The cache math follows from that.
  • MoE is not "a small model". The 26B A4B stores 25.2B parameters and activates 3.8B per token. Compute behaves like the active size; memory behaves like the stored size.
  • Modalities meet in the decoder. Text, image, and (on E2B/E4B) audio all become tokens in one sequence. Function calls are just structured tokens the runtime intercepts.

All numbers quoted here are from public docs and configs. Exact behavior at training time is not public; the model card calls out that architectural choices like hybrid attention, PLE, and MoE routing apply, but individual hyperparameters can differ between the public config and the internal training recipe.
